# UI源码迁移真实数据接入优化点

检查日期：2026-04-27

## 1. 当前结论

`07_UI源码迁移到TanStack_Cloudflare_Phase拆分与细化计划.md` 中大部分 Phase 已被勾选，但从源码看，计划尚未真正完成。

当前状态更准确地说是：

- TanStack Start + Cloudflare 工程底座已建立。
- UI 页面、路由、应用壳子和发票 intake / review 工作台已迁移。
- D1 schema、R2、Queue、Workers AI 的 binding 和部分底层管线已存在。
- 但 Phase 7 的“页面从本地 mock 切到真实 loader / Query / Form”没有完成。
- 多个核心页面仍依赖 `fallback-store`、浏览器 `localStorage` / `sessionStorage`、seed 数据和 demo 发票任务。

因此，这个项目目前不是“已全面接入真实数据库”，而是“UI 迁移完成度较高，真实数据层只完成了一部分骨架和发票上传管线”。

## 2. 已完成度判断

### 基本完成

- Phase 0：迁移边界和目录策略已明确。
- Phase 1：TanStack Start + Cloudflare 底座已建立。
- Phase 2：设计系统、全局样式和应用壳子已迁移。
- Phase 3：基础页面路由已建立，但仍有一个手动验证项未勾选。
- Phase 4：发票 intake 与 review 工作台已拆分。
- Phase 5：D1 schema、migration、query/mutation 文件边界已建立。
- Phase 6：R2 / Queue / Workers AI 异步链路骨架已建立。

### 未真正完成

- Phase 7：真实数据接线没有完成。
- Phase 8：构建、类型生成、测试可以通过，但测试通过并不代表真实数据库链路完整。

原因是现有测试主要覆盖 UI smoke、fallback store、hydration 和部分管线能力，没有充分验证生产业务页面是否全部读写 D1。

## 3. 主要 mock / fallback 来源

### 3.1 `fallback-store` 仍是核心业务数据源

文件：

- `src/lib/server/fallback-store.ts`

问题：

- `listStoredSalesRecords()` 会合并 seed 营业额数据。
- `listStoredInvoiceJobs()` 会合并 demo 发票任务。
- 营业额写入浏览器 `localStorage`。
- 发票任务写入浏览器 `sessionStorage`。
- 日历和分析数据依赖确定性 seed 数据，而不是 D1 聚合。

典型 mock 数据包括：

- `seed-sales-*`
- `demo-metro-apr`
- `demo-supplier-ready`
- `demo-bodega-ready`
- `系统初始化样例数据`

### 3.2 营业额没有接 D1

文件：

- `src/lib/server/queries/sales.ts`
- `src/lib/server/mutations/sales.ts`
- `src/routes/sales/new.tsx`

问题：

- 页面已经使用 TanStack Form。
- 但 query 仍读 `getStoredSalesRecord()` / `listStoredSalesRecords()`。
- mutation 仍写 `upsertStoredSalesRecord()`。
- 实际没有写入 `sales_daily` 表。

影响：

- 用户录入的营业额不会进入真实 D1。
- 换浏览器、清缓存、服务端渲染环境下数据不稳定。
- 首页、月分析、日历依赖的营业额数据不可信。

### 3.3 首页 summary 没有接 D1

文件：

- `src/lib/server/queries/dashboard.ts`

问题：

- 首页 summary 仍从 `fallback-store` 取营业额和发票任务。
- `pendingInvoiceCount`、`monthlyInvoiceCount`、`lastActivityLabel` 可能来自 demo 发票。
- `monthlyExpenseTotal` 间接依赖 mock 分析数据。

影响：

- 首页看起来有数据，但不一定来自真实业务库。

### 3.4 月分析和日历仍有确定性假数据

文件：

- `src/lib/server/queries/analytics.ts`
- `src/lib/server/fallback-store.ts`

问题：

- `getCalendarMonthBase()` 生成确定性日历收入和支出。
- `buildIncomeBreakdown()` 对没有真实营业额的日期按固定比例拆分 BBVA / CAIXA / EFECTIVO。
- `buildExpenseBreakdown()` 按固定比例拆分酒水采购、食材、人工、其他。
- 月趋势和周趋势基于这些合成数据。

影响：

- 虽然已经删除了 `Math.random()`，但数据仍是假数据。
- 分析页和日历页还没有真正基于 `sales_daily`、`invoices`、`invoice_items`、`ledger_entries` 聚合。

### 3.5 发票页面存在真实管线和 fallback 双路径

真实管线文件：

- `src/lib/server/upload.ts`
- `src/lib/server/queue.ts`
- `src/lib/server/extraction.ts`
- `src/lib/server/queries/invoices.rpc.ts`
- `src/lib/server/mutations/invoices.rpc.ts`

fallback 文件：

- `src/lib/server/queries/invoices.ts`
- `src/lib/server/mutations/invoices.ts`
- `src/lib/server/fallback-store.ts`

问题：

- `invoices/new` 会先判断 `pipelineEnabled`。
- 如果 D1/R2/Queue/schema 不完整，页面会退回本地 fallback。
- fallback 会创建本地 fake intake job。
- “打开最近任务”仍有 `demo-metro-apr` 兜底。

影响：

- 本地开发或 binding 不完整时，用户看到的是 mock 行为。
- 生产环境如果 schema/readiness 检查失败，也可能出现非预期降级。

### 3.6 发票确认没有真正入账

文件：

- `src/lib/server/mutations/invoices.rpc.ts`
- `migrations/0001_initial.sql`
- `src/lib/db/schema.ts`

问题：

- schema 已有 `invoices`、`invoice_items`、`ledger_entries`。
- 但确认发票时主要更新 `extraction_results.structured_json` 和 `intake_jobs.stage`。
- 没有把确认后的 header 写入 `invoices`。
- 没有把 line items 写入 `invoice_items`。
- 没有生成 `ledger_entries`。

影响：

- “确认入账”只是改变 review job 状态。
- 后续真实成本分析、供应商分析、原料价格分析都缺少规范化数据源。

### 3.7 原料选项仍是硬编码

文件：

- `src/lib/server/app-domain.ts`

问题：

- `ingredientOptions` 是硬编码数组。
- 没有从 `ingredients` 和 `ingredient_aliases` 表查询。
- 发票行项目映射不是真实原料库映射。

影响：

- 原料库、别名、供应商映射无法真实维护。
- 后续成本诊断无法依赖该映射结果。

### 3.8 测试仍偏向 mock 验证

文件：

- `src/tests/invoice-mock-store.test.ts`
- `src/tests/router.smoke.test.tsx`
- `src/tests/invoice-intake-rehydration.test.tsx`
- `src/tests/invoice-review-rehydration.test.tsx`

问题：

- 测试仍覆盖 mock store。
- smoke test 直接使用 `demo-metro-apr`。
- 现有测试能保证 fallback 行为不坏，但不能证明真实 D1 业务链路完整。

影响：

- Phase 8 的测试通过容易造成“真实数据已完成”的错觉。

## 4. 优先优化项

### P0: 明确禁用生产 fallback

目标：

- 生产环境不能静默使用 mock / fallback 数据。

建议：

- 引入显式配置，例如 `ENABLE_DEMO_DATA` 或 `VITE_ENABLE_DEMO_DATA`。
- 只有本地开发和测试环境允许 fallback。
- 生产环境缺少 DB/R2/Queue/schema 时直接报错或显示明确的系统配置错误。
- 把 `pipelineEnabled ? real : fallback` 改为更严格的环境判断。

验收标准：

- 生产构建中业务页面不会 import 或执行 `fallback-store` 数据路径。
- 搜索 `fallback-store` 时，引用只剩测试、开发 demo 或显式 demo 模块。

### P0: 营业额读写接入 D1

目标：

- `/sales/new` 真实读写 `sales_daily`。

建议：

- 新增 sales server functions，例如：
  - `getSalesEntryPageDataServerFn`
  - `saveSalesDraftServerFn`
  - `submitSalesEntryServerFn`
- mutation 写入 `sales_daily`，按 date 做 upsert。
- dashboard、analytics、calendar 统一读取 D1 中的 `sales_daily`。

验收标准：

- 提交营业额后，D1 `sales_daily` 有对应记录。
- 换浏览器后仍能看到已提交数据。
- 清空 localStorage 不影响真实数据。

### P0: 分析页和日历页接入真实聚合

目标：

- `/analytics/monthly` 和 `/analytics/calendar` 不再使用确定性假数据。

建议：

- 收入从 `sales_daily` 聚合。
- 支出优先从 `ledger_entries` 聚合。
- 发票确认前可使用 `intake_jobs` / `extraction_results` 做待核对统计，但不应混入正式支出。
- 支出结构从 `ledger_entries.category` 或 `invoice_items` / `ingredients.category` 聚合。

验收标准：

- 没有真实数据时，页面显示 0 或空状态，而不是自动生成数据。
- `getCalendarMonthBase()` 不再作为生产数据源。
- `buildExpenseBreakdown()` 不再使用固定比例。

### P0: 发票确认真正写入业务表

目标：

- 点击“确认入账”后，生成真实 `invoices`、`invoice_items`、`ledger_entries`。

建议：

- 在 `confirmInvoiceReviewJobServerFn` 中增加事务式写入逻辑。
- 根据 review header 写入 `invoices`。
- 根据 line items 写入 `invoice_items`。
- 根据发票总额生成采购类 `ledger_entries`。
- `intake_jobs.stage` 更新为 `ready`。
- 避免重复确认导致重复入账，可用 `intake_job_id` 做幂等键。

验收标准：

- 确认发票后，D1 中存在对应 invoice。
- line items 被规范化保存。
- 月分析支出能反映这张发票。
- 重复点击确认不会重复生成 ledger entry。

### P1: 原料库接 D1

目标：

- review table 的原料下拉来自真实 `ingredients`。

建议：

- 新增 `src/lib/server/queries/ingredients.ts`。
- 读取 `ingredients` 表作为 `ingredientOptions`。
- 后续支持 `ingredient_aliases` 做供应商商品名自动匹配。
- 提供 seed migration 或后台录入入口，初始化常用原料。

验收标准：

- `ingredientOptions` 不再是硬编码数组。
- 新增原料后，review 页面可选择。
- 发票行项目保存的是真实 `ingredient_id`。

### P1: 发票 review 页面展示真实文件预览

目标：

- `DocumentPreview` 不只是 UI 占位，要能访问 R2 中的原始文件或安全预览 URL。

建议：

- 增加只读预览 endpoint / server function。
- 根据 `source_documents.r2_key` 读取 R2 文件。
- PDF 和图片分别处理。

验收标准：

- review 页面能看到上传的真实文件。
- jobId 不存在或无权限时不能读取文件。

### P1: 替换 demo 测试为真实数据测试

目标：

- 测试能防止真实数据路径回退到 mock。

建议：

- 保留少量 fallback 单元测试，但标注为 dev/test-only。
- 增加 D1 mock/miniflare 或 adapter 级测试。
- 添加静态约束测试：生产 query/mutation 不允许 import `fallback-store`。
- smoke test 不再依赖 `demo-metro-apr`。

验收标准：

- 测试覆盖：
  - sales_daily upsert
  - dashboard summary from D1
  - monthly analytics from D1
  - invoice upload creates source_documents + intake_jobs
  - confirm invoice writes invoices + invoice_items + ledger_entries

### P2: 更新计划文档状态

目标：

- 让文档反映真实完成度，避免误判。

建议：

- 把 Phase 7 改成未完成或部分完成。
- Phase 8 标注为“构建与资源配置完成，但真实数据切换未完成”。
- 把 mock/fallback 清理作为新的 Phase 9 或补充任务。

验收标准：

- 文档状态与源码事实一致。
- 后续开发者不会误以为真实数据库链路已经完成。

## 5. 建议执行顺序

1. 先禁用生产 fallback，并明确 demo 数据开关。
2. 接入 `sales_daily` 的 D1 读写。
3. 用 D1 重写 dashboard、monthly analytics、calendar analytics。
4. 完成发票确认入账，写入 `invoices`、`invoice_items`、`ledger_entries`。
5. 把原料选项改为 D1 查询。
6. 清理 demo job、seed sales、mock store 测试依赖。
7. 更新 `07` 计划文档中的 Phase 状态。

## 6. 最小完成定义

只有满足以下条件，才可以认为 `07` 计划中的真实数据切换完成：

- 首页 summary 来自 D1。
- 营业额录入读写 `sales_daily`。
- 发票上传写 R2、`source_documents`、`intake_jobs` 并进入 Queue。
- 发票 review 读取 D1 中的 intake job 和 extraction result。
- 发票确认写入 `invoices`、`invoice_items`、`ledger_entries`。
- 月分析和日历页基于 D1 聚合。
- 生产环境不会静默回落到 `fallback-store`。
- 测试覆盖真实数据路径，而不只覆盖 mock / fallback。
