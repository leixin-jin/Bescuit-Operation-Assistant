## 1. 这份 V2 计划解决什么问题

这次不是继续做一个“大而全的餐饮成本管理系统”。
而是先把范围纠正成一个真正能落地的第一版：

**一个服务于老板自用的“采购成本追踪 + 原料标准化 + 后续毛利打底”的轻量系统。**

它先解决 4 件事：

1. **让发票稳定回流到我手里，并能批量录入**
2. **把最近最该盯的前 10 个高支出原料标准化**
3. **基于标准原料看价格走势、比价和替代采购判断**
4. **为未来 10 道主卖菜的理论毛利计算打基础**

一句话：

**V2 的重点不是“把所有东西都录进去”，而是“把最该管的东西先管起来”。**

## 2. 与旧版开发计划相比，核心纠偏是什么

| 主题 | 旧版思路 | V2 调整后 |
|---|---|---|
| 系统定位 | 餐饮成本管理 Web App | 老板自用的采购成本追踪工具 |
| 录入方式 | 以“上传 JSON”页面为中心 | 以“发票回流 SOP + 批量录入工作台”为中心 |
| 分析对象 | 直接按 `product_name` 分析 | 先映射到**标准原料主表**再分析 |
| 仪表盘范围 | 一开始就做广泛 dashboard | 先只做**前 10 个重点原料** |
| 比价逻辑 | 直接比较原始商品名单价 | 统一单位、规格后比较标准单位成本 |
| 经营动作 | 偏展示 | 支持“换规格 / 换品牌”的替代采购决策 |
| 菜品毛利 | 默认很快就能做 | 延后到“10 道主卖菜近似配方卡”准备好之后 |
| 员工参与方式 | 容易滑向员工端上传 | 员工只负责把发票放进固定柜子 |

## 3. V2 的正确问题定义

V2 第一阶段不回答这些问题：

- 全部菜品到底赚不赚钱
- 浪费、损耗、偷漏到底发生在哪
- 整家店的完整经营利润是多少
- 员工如何在手机上直接录单

V2 第一阶段只回答这些问题：

1. **最近 30 天采购金额最高的前 10 个原料是什么？**
2. **这些原料的标准单位成本怎么变化？**
3. **哪些原料最近超出价格阈值，需要换规格 / 换品牌？**
4. **哪些基础数据已经足够支持后续做 10 道主卖菜的理论毛利？**

## 4. 第一版范围：做什么，不做什么

### 4.1 第一版必须做

#### A. 发票回流与批量录入
- 固定门店唯一入口：发票柜 / 发票盒 / 发票抽屉
- 老板每日 1 次或 2 次统一收柜
- Gemini 提取 JSON 后，进入系统的**批量录入工作台**
- 单据有问题时先进入 **draft 队列**，不阻塞整批录入

#### B. 标准原料主表
- 只先服务于前 10 个高支出原料
- 为每个原料定义：
  - 标准名称
  - 标准比较单位（kg / L / ud）
  - 常见规格
  - 单位换算方式
  - 常见别名 / 供应商写法
  - 可接受价格区间

#### C. 替代采购规则
- 记录每个重点原料的：
  - 可接受替代品牌
  - 可接受替代规格
  - 不能下降的质量底线
  - 超价时的动作建议

#### D. 成本趋势与替代采购看板
- 只针对重点原料 / 分类
- 看：
  - 最近价格走势
  - 最近供应商报价比较
  - 超阈值提醒
  - 替代采购建议

#### E. 为理论毛利做基础准备
- 预留“10 道主卖菜近似配方卡”数据结构
- 但第一版**不急着做完整毛利系统**

### 4.2 第一版明确不做

- 员工端录入 App
- App 内置 OCR
- 全菜单实时毛利分析
- 损耗 / 偷漏 / 稽核系统
- 复杂库存系统
- 高精度出成率建模
- 为所有原料做一次性完整标准化

一句话：

**第一版只做“老板每天会真正用”的最小系统。**

> 说明：预算模块不是当前重点，后续如果你真的需要，再单独规划。

## 5. 核心业务流程（按真实门店执行方式设计）

```text
供应商送货 / 店员收到单据
        ↓
员工只做一个动作：把发票放进固定柜子
        ↓
我每天固定时间收柜
        ↓
我统一拍照 / 用 Gemini 提取 JSON
        ↓
我把 JSON 粘贴到批量录入工作台
        ↓
系统校验并生成“待确认批次”
        ↓
能自动映射的原料直接入库
不能映射的原料进入 draft / 待标准化队列
        ↓
我完成标准原料映射与单位换算
        ↓
系统按标准原料更新趋势、比价与替代采购判断
```

这里最重要的不是“上传页面长什么样”。
而是两件事：

1. **发票先完整回流**
2. **原料先被稳定映射成标准原料**

没有这两步，后面的图都不可信。

## 6. 产品结构（V2）

### 6.1 页面 / 路由建议

| 路径 | 页面 | 作用 |
|---|---|---|
| `/` | 仪表盘 | 看本月重点原料支出、价格提醒、待处理情况 |
| `/upload` | 批量录入工作台 | 粘贴 Gemini JSON、生成待处理批次 |
| `/drafts` | 待处理队列 | 查看未映射、单位不清、金额异常的记录 |
| `/ingredients` | 标准原料主表 | 管理标准名称、单位、别名、规格换算、阈值 |
| `/analytics` | 原料分析 | 查看重点原料价格走势、供应商比较 |
| `/suppliers` | 供应商管理 | 管理常见供应商信息 |
| `/recipes` | 近似配方卡（后续阶段） | 只服务 10 道主卖菜 |

### 6.2 第一版最关键的 4 个模块

#### 模块 1：批量录入工作台
目标不是单张录入优雅，而是**整批吞吐量高**。

需要支持：
- 一次粘贴一张或多张 JSON
- 批量预览
- 批量保存
- 保存为 draft
- 键盘优先操作

#### 模块 2：标准原料主表
这是第一版的中枢。

没有这个模块，就会回到：
- `Tomate`
- `Tomate pera`
- `Caja tomate 6kg`
- `Tomate 1kg`

看起来像 4 个东西，实际上可能是同一个管理对象。

#### 模块 3：映射与标准化队列
系统必须允许：
- 先收原始数据
- 再逐步标准化

而不是要求：
- 每一张发票进来时就 100% 干净

#### 模块 4：重点原料看板
只看最关键的少数：
- 前 10 原料采购金额排行
- 单位成本走势
- 最近供应商报价
- 超价提醒
- 替代采购建议

## 7. 数据模型重构（重点纠偏）

旧版最大的问题，是把原始 `product_name` 直接当成分析主键。
V2 必须改成：

**原始行项目 ≠ 标准原料**

### 7.1 推荐核心表

#### 1. `suppliers`
供应商表。

#### 2. `invoices`
发票 / 送货单主表。

#### 3. `invoice_items`
保存原始行项目，但新增标准化相关字段：
- `raw_product_name`
- `raw_quantity`
- `raw_unit`
- `raw_unit_price`
- `ingredient_id`（可空）
- `normalized_quantity`
- `normalized_unit`
- `normalized_unit_price`
- `mapping_status`（pending / mapped / draft）

> 重点：**这张表既保存原始事实，也保存映射结果。**

#### 4. `ingredients`
标准原料主表：
- 标准名称
- 分类
- 标准单位
- 是否重点原料
- 价格下限 / 上限
- 默认备注

#### 5. `ingredient_aliases`
原料别名映射表：
- 供应商原始写法
- 对应标准原料
- 规格说明
- 默认换算规则

#### 6. `ingredient_substitution_rules`
替代采购规则：
- 标准原料
- 可接受品牌 / 规格
- 质量底线
- 超价时建议动作

#### 7. `recipe_cards`（后续阶段）
近似标准配方卡：
- 菜名
- 售价
- 固定容器 / 摆盘
- 主要原料
- 每份大致用量
- 波动备注

### 7.2 一个关键原则

所有分析逻辑，必须尽量基于：
- `ingredient_id`
- `normalized_unit`
- `normalized_unit_price`

而不是基于：
- 原始 `product_name`
- 原始 `unit_price`

否则 dashboard 只是把脏数据画成图。

## 8. 关键功能设计（V2）

### 8.1 批量录入工作台

#### 目标
让“我自己每天固定收柜后统一录入”这件事变快。

#### 需要的交互
- 粘贴 JSON 后先做结构校验
- 生成批次预览
- 对每张单据打状态：
  - 可直接入库
  - 需要补字段
  - 需要映射原料
  - 需要人工复核金额
- 支持“全部保存为 draft”

#### 第一版不要追求的事
- 员工手机拍照直传
- App 内直接 OCR
- 特别复杂的逐格编辑体验

### 8.2 draft 队列

#### 触发条件
以下情况都可以进 draft：
- 供应商不确定
- 商品名没见过
- 单位无法换算
- 金额异常
- 当前不确定是否属于重点原料

#### 设计原则
- **不要让一张坏单卡住整批录入**
- 先收进去，再慢慢修

### 8.3 标准原料主表

每个重点原料至少要有：
- 标准名称
- 分类
- 标准单位
- 原始别名
- 常见规格
- 换算规则
- 可接受价格区间
- 是否为前 10 重点原料

### 8.4 价格走势与比价

V2 不再做“商品名搜索后直接画图”的简单逻辑。
而是：
- 先选标准原料
- 再按标准单位查看价格走势
- 再比较最近不同供应商报价

这样看见的才不是幻觉。

### 8.5 替代采购支持

系统要支持你最常见的动作：
- 换规格
- 换品牌

所以第一版要能看到：
- 当前原料最近价格是否异常
- 当前供应商是不是明显更贵
- 有没有预设的可替代规格 / 品牌
- 当前是否触发“该换”的判断

### 8.6 菜品理论毛利（放到后续阶段）

只有当下面这些条件成立后，才进入：
- 10 道主卖菜被选出来
- 每道菜有售价
- 每道菜有固定容器 / 固定摆盘
- 每道菜能给出近似标准配方
- 重点原料单位成本已经可计算

在此之前，不做“全菜单毛利看板”。

## 9. 第一版仪表盘应该看什么

### 必须看
- 本月总采购金额
- 前 10 原料采购金额排行
- 前 10 原料单位成本走势
- 最近超价提醒
- 最近 5 张已录入 / 待处理发票

### 可以有，但不是第一优先级
- 分类支出分布
- 供应商占比
- 最近价格波动最大的原料

### 暂时不要放
- 菜品利润排行榜
- 复杂库存图表
- 损耗 / 偷漏指标
- 看起来很忙但没动作指向的统计卡片

## 11. 推荐数据库字段方向

下面不是最终 SQL，而是 V2 应该坚持的结构方向。

### `invoice_items`
| 字段 | 说明 |
|---|---|

| `raw_unit_price` | 原始单价 |

| `normalized_unit` | 换算后的标准单位 |
| `normalized_unit_price` | 标准单位成本 |

### `ingredients`
| 字段 | 说明 |
|---|---|
| `name` | 标准原料名 |

| `base_unit` | 标准单位 |

| `price_lower_bound` | 合理低价 |
| `price_upper_bound` | 合理高价 |

| 字段 | 说明 |
|---|---|

## 12. 分析逻辑要怎么改

### 旧逻辑
- 直接按 `product_name` 搜索
- 直接对原始 `unit_price` 做走势
- 直接做供应商比价

### 新逻辑
- 先找到 `ingredient_id`
- 统一到标准单位
- 用 `normalized_unit_price` 做走势
- 用最近一次有效映射记录做供应商比较
- 结合 `ingredient_substitution_rules` 给出“是否需要换规格 / 换品牌”的动作提示

这一步很关键。

**V2 不是把图做得更漂亮，而是把分析主键从“原始商品名”纠正为“标准原料”。**

## 13. 验收标准（V2 是否真的有用）

第一版上线后，至少要满足以下标准：

### A. 发票录入层面
- 我可以每天固定时间处理一批发票
- 单张异常不会卡住整批
- 所有发票都能进入“已录入 / draft / 待映射”之一

### B. 原料标准化层面
- 前 10 个高支出原料都有标准主表记录
- 常见别名能映射回标准原料
- 同一原料能统一成同一种比较单位

### C. 管理层面
- 我能看出前 10 个原料最近的单位成本走势
- 我能知道某个重点原料最近该不该换规格 / 换品牌
- 我能根据替代采购规则做出下一步动作

### D. 后续扩展层面
- 系统已经有足够数据结构，支撑后面做 10 道主卖菜近似理论毛利

如果做不到这些，就说明系统还停留在“录入器”阶段，还没进入“管理工具”阶段。

## 14. 技术栈保持不变，但实现顺序要变

### 保持不变
- TanStack Start
- Cloudflare Workers
- Cloudflare D1
- Drizzle ORM
- Gemini 手工提取 JSON

### 要改变的不是技术，而是顺序
旧顺序：
1. 先录入
2. 先 dashboard
3. 再想怎么管理

V2 顺序：
1. 先回流
2. 先标准化
3. 先聚焦前 10 原料
4. 再做 dashboard
5. 再逐步进入近似毛利

> 预算模块暂时不纳入当前版本，后续如果要做，再单独规划。

## 15. 一句话结论

**它应该做成一个：让我自己每天能批量录入发票、稳定追踪前 10 个重点原料、支持替代采购判断、并为后续 10 道主卖菜毛利打基础的老板自用工具。**
