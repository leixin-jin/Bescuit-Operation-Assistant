# 营业额登记与可视化

## 目标

这份文档只保留你现在真正需要接到酒吧成本计划 APP 的三页功能：

| 工作表 | 作用 | 说明 |
| --- | --- | --- |
| `日记账` | 原始录入层 | 所有流水都从这里进入，是唯一事实数据源。 |
| `月度` | 月度总览 | 顶部 KPI 卡片 + 当月收入/支出结构图。 |
| `日历` | 日级视图 | 月历格子里显示每日收入/支出，并带日趋势图。 |

简化后的目标很明确：

- 用户输入营业额
- 用户输入发票/支出
- 系统自动生成月报和日历报表

这意味着第一版不再保留 `全年`、`账户`、`支出预算`、`家庭成员` 这些页面与逻辑。

## 简化后的报表架构

保留后的核心结构只有一条链路：

```text
营业额录入 / 发票录入
        ->
      日记账
        ->
  日度聚合 + 月度聚合
        ->
   月度页 + 日历页
```

在 APP 里，`日记账` 是唯一事实表，`月度` 和 `日历` 都应该从它聚合出来，而不是各自单独存数。

## 日记账

### 页面职责

`日记账` 只做一件事：承接所有原始流水。

它不是报表页，而是报表的唯一数据来源。

### 建议保留的字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `date` | 是 | 交易日期 |
| `type` | 是 | `收入` 或 `支出` |
| `category` | 是 | 收入渠道或支出类别 |
| `amount` | 是 | 金额 |
| `account` | 否 | 付款/收款账户，V1 可留作辅助字段 |
| `vendor` | 否 | 发票供应商，支出时常用 |
| `note` | 否 | 备注 |
| `source_type` | 是 | `sales` 或 `invoice` |
| `source_id` | 否 | 对应来源记录 ID |

### 营业额录入逻辑

酒吧场景下，收入本质上更像“收款渠道拆分”，不是复杂的收入品类。

建议保留这 3 个渠道：

- `BBVA`
- `CAIXA`
- `EFECTIVO`

推荐两种录入方式：

1. 直接录入三个渠道金额
2. 录入 `总营业额 + 已知渠道金额`，系统自动反推 `EFECTIVO`

无论采用哪种方式，最终都应该落成 `日记账` 里的 `收入` 记录。

### 发票录入逻辑

发票录入最终也要落成 `日记账` 里的 `支出` 记录。

建议支出记录至少包含：

- 日期
- 供应商
- 支出类别
- 金额
- 支付方式
- 发票附件

## 保留的核心 KPI

简化后只保留 6 个最有用的指标。

### 1. 月收入

```text
月收入 = SUM(金额 where type = 收入 and 日期属于选中月份)
```

### 2. 月支出

```text
月支出 = SUM(金额 where type = 支出 and 日期属于选中月份)
```

### 3. 月净额

```text
月净额 = 月收入 - 月支出
```

这个指标比原模板里的“转入/转出/还款/结余”更适合你的第一版 APP，因为它直接回答一个问题：

这个月赚进来多少，花出去多少，净流入多少。

### 4. 收入结构占比

按收款渠道看当月收入占比：

- `BBVA`
- `CAIXA`
- `EFECTIVO`

### 5. 支出结构占比

按支出类别看当月支出占比。

### 6. 每日收入 / 每日支出

按天汇总：

```text
日收入 = SUM(金额 where type = 收入 and 日期 = 当天)
日支出 = SUM(金额 where type = 支出 and 日期 = 当天)
```

这组数据同时给 `日历` 页面和日趋势图使用。

## 月度页

### 页面职责

`月度` 是经营者每个月打开后第一眼看到的总览页。

它只回答三类问题：

- 这个月收入多少
- 这个月支出多少
- 钱主要从哪里来，又主要花到哪里去

### 页面结构

建议保留 3 张 KPI 卡片：

- `月收入`
- `月支出`
- `月净额`

下面保留 2 张结构图：

- 左侧：`当月支出结构图`
- 右侧：`当月收入结构图`

### 可视化逻辑

#### 1. 顶部 KPI 卡片

数据全部来自当月聚合结果：

- `月收入`
- `月支出`
- `月净额`

#### 2. 当月收入结构图

推荐使用饼图或环形图。

维度：

- `BBVA`
- `CAIXA`
- `EFECTIVO`

规则：

- 只显示金额大于 0 的渠道
- 默认按金额从高到低排序

#### 3. 当月支出结构图

推荐使用饼图或环形图。

维度：

- 各支出类别

规则：

- 只显示金额大于 0 的类别
- 默认按金额从高到低排序
- 类别过多时，尾部可以合并成 `其他`

### 交互建议

- 页面顶部只有一个月份选择器
- 所有卡片和图表都跟随同一个月份切换
- 不要让不同区域各自选时间，否则用户会觉得报表对不上

## 日历页

### 页面职责

`日历` 是用来观察月内波动的。

它回答的问题不是“这个月总共多少”，而是：

- 哪几天收入高
- 哪几天支出高
- 是否存在异常波动

### 页面结构

保留两层视觉：

1. 月历格子
2. 日趋势图

### 可视化逻辑

#### 1. 月历格子

每一天的格子里显示：

- 当日收入
- 当日支出

建议样式：

- 收入用一种颜色
- 支出用另一种颜色
- 没有数据的日期显示为空或 `0`

#### 2. 日趋势图

推荐使用双折线图或双柱图。

两条序列分别是：

- `每日收入`
- `每日支出`

横轴是当月的 1 到最后一天。

### 交互建议

- `日历` 页和 `月度` 页共用同一个月份筛选口径
- 点击某一天时，可以下钻查看当天的营业额和发票明细
- 如果第一版不做下钻，至少要保证格子和趋势图的数据完全一致

## 聚合逻辑

这三个页面需要的计算其实很简单，核心只有 4 组：

### 月度聚合

```text
month_revenue = sum(amount where type = 收入 and month = selected_month)
month_expense = sum(amount where type = 支出 and month = selected_month)
month_net = month_revenue - month_expense
```

### 月度收入结构

```text
group by category
where type = 收入 and month = selected_month
sum(amount)
```

### 月度支出结构

```text
group by category
where type = 支出 and month = selected_month
sum(amount)
```

### 日历数据

```text
group by date
where date in selected_month
sum(income_amount), sum(expense_amount)
```

## 接入 APP 的最小方案

### 输入端

V1 只做两个入口就够了：

1. `营业额录入`
2. `发票录入`

### 数据层

V1 最简单的做法是：

- 保留业务输入表：`sales_daily`、`expense_invoices`
- 同时统一生成一张事实表：`ledger_entries`

其中 `ledger_entries` 作为报表唯一来源。

### 输出端

系统自动生成两张报表页：

1. `月度`
2. `日历`

`日记账` 则作为后台明细页或管理页存在。

## 自动生成逻辑

### 营业额录入后

系统执行：

- 写入 `sales_daily`
- 生成对应的 `收入` 日记账记录
- 刷新 `月度` KPI
- 刷新 `月度` 收入结构图
- 刷新 `日历` 当日数据
- 刷新 `日趋势图`

### 发票录入后

系统执行：

- 写入 `expense_invoices`
- 生成对应的 `支出` 日记账记录
- 刷新 `月度` KPI
- 刷新 `月度` 支出结构图
- 刷新 `日历` 当日数据
- 刷新 `日趋势图`

## 不再保留的内容

这次简化后，以下内容从产品第一版里移除：

- `全年` 页
- `账户` 页
- `支出预算` 页
- `家庭成员` 维度
- `转入 / 转出 / 还款` 相关 KPI

如果未来要做第二版，再把预算和账户现金流加回来会更合理。

## 结论

这份报表简化后，实际上已经变成一个非常清晰的 V1：

- `日记账` 负责存事实
- `月度` 负责看当月经营结果和结构
- `日历` 负责看每天波动

对你的酒吧成本计划 APP 来说，这样的范围最稳，也最适合先落地“输入发票 + 输入营业额 -> 自动生成报表”这条主链路。
